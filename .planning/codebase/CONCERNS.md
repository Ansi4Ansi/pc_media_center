# PC Media Center — Potential Concerns & Questions

> Generated by GSD Codebase Mapper (Concerns Focus)

## Overview

This document identifies potential issues, questions, and areas for improvement in the PC Media Center codebase.

---

## Architecture Concerns

### 1. Domain Layer Dependencies
**Potential Issue**: The domain layer should have zero dependencies, but verify that:
- No imports from `data/` or `presentation/`
- All repository interfaces are pure Dart classes
- Use cases don't import Flutter-specific packages

**Verification Needed**: Run static analysis to confirm no cross-layer imports.

---

### 2. BLoC State Complexity
**Potential Issue**: As the app grows, state objects may become complex:
- Large state trees can be hard to debug
- Deep nesting makes UI updates difficult

**Recommendation**: Consider splitting large states into smaller, focused blocs.

---

### 3. Repository Interface Coupling
**Potential Issue**: Multiple repositories depending on each other through interfaces may create tight coupling in the domain layer.

**Recommendation**: Review repository dependencies to ensure clean separation.

---

## Database Concerns

### 1. Migration Strategy
**Current State**: Drift handles migrations automatically.

**Potential Issue**: 
- Large schema changes may require data migration logic
- Backward compatibility with existing database files

**Recommendation**: Document major schema changes and test migration paths.

---

### 2. Database Location
**Current**: `data/media_center.db` next to executable.

**Potential Issue**:
- Database file grows over time (cached images, search results)
- No automatic cleanup mechanism mentioned
- Large database may impact performance

**Recommendation**: Implement periodic cleanup of cached data with user-configurable limits.

---

## API Integration Concerns

### 1. API Key Management
**Current**: User-provided API keys stored in settings table.

**Potential Issues**:
- Keys stored locally may be exposed if device is compromised
- No key rotation mechanism
- Single point of failure if one API changes terms

**Recommendation**: Consider encrypting sensitive data at rest or using secure storage APIs.

---

### 2. Offline Behavior
**Current**: App works offline with cached data.

**Potential Issue**: 
- Cached API responses may become stale
- No clear strategy for detecting and refreshing stale cache
- User may not be aware of outdated information

**Recommendation**: Implement cache freshness indicators and automatic refresh prompts.

---

## Platform-Specific Concerns

### 1. Icon Extraction Reliability
**Current**: Platform-specific extraction with fallback to placeholders.

**Potential Issues**:
- Some executables may not have extractable icons
- Different icon sizes across platforms
- Fallback icons may look inconsistent

**Recommendation**: Test icon extraction on various applications and ensure consistent fallback behavior.

---

### 2. File Launching Edge Cases
**Current**: Uses OS-specific commands for file launching.

**Potential Issues**:
- Some files have no associated application
- User may not want to open certain file types
- Permission errors when accessing protected directories

**Recommendation**: Add user confirmation dialogs and configurable allowed file types.

---

## Performance Concerns

### 1. Image Caching Strategy
**Current**: `cached_network_image` for poster caching.

**Potential Issues**:
- Unbounded cache growth over time
- No clear eviction policy mentioned
- Large number of posters may consume significant disk space

**Recommendation**: Implement LRU cache with size limits and periodic cleanup.

---

### 2. Search Performance
**Current**: Online search via TMDB/Kinopoisk APIs.

**Potential Issues**:
- API rate limits may be hit during heavy usage
- Network latency affects user experience
- No local search index for offline queries

**Recommendation**: Implement local search index and request debouncing.

---

## Security Concerns

### 1. Local Data Exposure
**Potential Issues**:
- Database file readable by any user with access to app directory
- Cached images and metadata stored in plaintext
- No encryption of sensitive user data

**Recommendation**: Consider encrypting database or using platform-specific secure storage.

---

### 2. API Key Exposure
**Current**: Keys stored in plain text in settings table.

**Potential Issue**: If device is compromised, API keys are exposed.

**Recommendation**: Use encrypted preferences or secure keychain/storage APIs.

---

## Testing Concerns

### 1. Integration Test Coverage
**Current Status**: Widget tests exist; integration tests mentioned but not detailed.

**Potential Issue**: Limited end-to-end test coverage may miss critical bugs in user flows.

**Recommendation**: Implement key user flow integration tests (add item, search, launch).

---

### 2. BLoC Test Coverage
**Current**: `bloc_test` library available.

**Potential Issue**: As BLoCs grow, manual testing becomes tedious.

**Recommendation**: Write comprehensive state transition tests for each BLoC.

## User Experience Concerns

### 1. Error Handling
**Potential Issues**:
- Unclear error messages when API fails
- No retry mechanism for failed operations
- Graceful degradation not well-defined

**Recommendation**: Implement user-friendly error messages and automatic retries with backoff.

---

### 2. Loading States
**Current**: Loading indicators mentioned in widgets.

**Potential Issue**: Inconsistent loading states across the app may confuse users.

**Recommendation**: Standardize loading patterns and consider skeleton screens for better perceived performance.

---

## Accessibility Concerns

### 1. Keyboard Navigation
**Current**: `FocusableWidget` wrapper for keyboard support.

**Potential Issues**:
- Focus trapping not well-tested across all screens
- Tab order may be unintuitive in complex layouts
- Screen reader compatibility unknown

**Recommendation**: Test with screen readers and ensure logical focus flow.

---

### 2. Color Contrast
**Current**: Dark and light themes implemented.

**Potential Issue**: May not meet WCAG AA/AAA contrast requirements for all UI elements.

**Recommendation**: Run automated accessibility audits and adjust colors as needed.
