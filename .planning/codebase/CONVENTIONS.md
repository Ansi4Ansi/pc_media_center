# PC Media Center — Code Conventions & Testing

> Generated by GSD Codebase Mapper (Quality Focus)

## Overview

PC Media Center follows strict coding conventions and uses comprehensive testing strategies.

---

## General Coding Rules

### Language & Comments
- **Code language**: English
- **Comments**: English, explaining "why" not just "what"
- **UI strings**: Always use i18n (ARB files), never hardcode text

### State Management Pattern
| Component | File Structure |
|-----------|----------------|
| BLoC | `*_bloc.dart` — logic |
| Events | `*_event.dart` — event definitions |
| States | `*_state.dart` — state definitions |

Use `Equatable` for events and states. Use sealed classes for states when using Dart 3+.

---

## Entities vs Models

| Type | Location | Purpose |
|------|----------|--------|
| **Entity** | `domain/entities/` | Business object, no serialization methods |
| **Model** | `data/models/` | DTO with fromJson/toJson, maps to Entity |

### Example Flow
```
API Response → Model (fromJson) → Entity (business logic)
```

---

## Use Case Pattern

- **One use case = one operation**
- **Method name**: `call()` for convenient invocation
- **Location**: `domain/usecases/<feature>/`

Example:
```dart
// domain/usecases/items/add_item.dart
class AddItem {
  Future<void> call({required Item item, required Category category});
}
```

---

## Widget Organization

| Layer | Location | Rules |
|-------|----------|-------|
| **Screens** | `presentation/screens/<feature>/` | Minimal logic, use BLoC for state |
| **Widgets** | `presentation/widgets/` | Reusable components, no direct BLoC access |
| **Common** | `presentation/widgets/common/` | Navigation wrappers, loading indicators |

### Widget Rules
- Widgets receive data via parameters (not BLoC references)
- Use `FocusableWidget` wrapper for all interactive elements
- Support keyboard navigation with arrow keys and Tab

## Database Conventions

| Aspect | Convention |
|--------|------------|
| **Database file location** | Next to executable: `data/media_center.db` |
| **Migrations** | Handled automatically by Drift |
| **Schema changes** | Add new columns, don't modify existing ones |
| **Foreign keys** | Use ON DELETE CASCADE for child tables |

## Testing Strategy

### Test Types

| Type | Location | Purpose |
|------|----------|--------|
| **Unit tests** | `test/domain/`, `test/core/` | Pure logic, no UI dependencies |
| **Widget tests** | `test/presentation/screens/` | Screen rendering and basic interactions |
| **BLoC tests** | `test/presentation/blocs/` | State transitions and event handling |
| **Integration tests** | `integration_test/` | End-to-end user flows |

### Testing Libraries

| Library | Purpose |
|---------|--------|
| **flutter_test** | Flutter's built-in testing framework |
| **bloc_test** | BLoC-specific unit tests with proper lifecycle |
| **mocktail** v1.0.4 | Type-safe mocking (Dart 3+) |

### Mocking Strategy
```dart
// Use mocktail for clean, type-safe mocks
class MockCategoryRepository extends Mock implements CategoryRepository {
}
```

## BLoC Structure Convention

### File Naming
| File | Purpose |
|------|--------|
| `category_bloc.dart` | State management logic |
| `category_event.dart` | Event definitions (extends BaseEvent) |
| `category_state.dart` | State definitions (extends BaseState) |

### Example Structure
```
domain/usecases/
└── categories/
    ├── get_categories.dart      # Use case interface
    └── get_categories_impl.dart # Implementation

presentation/blocs/category/
├── category_bloc.dart           # BLoC logic
├── category_event.dart          # Events
└── category_state.dart          # States
data/repositories/
└── category_repository_impl.dart # Repository implementation
```

## Error Handling Conventions

### Failure Classes
| Location | Purpose |
|----------|--------|
| `core/errors/failures.dart` | BaseFailure and specific failure types |
| `core/errors/exceptions.dart` | Custom exceptions |

### Failure Types
- `DataFailure`: Network errors, API failures
- `DatabaseFailure`: Database operation errors
- `ValidationError`: Invalid input data
- `PermissionFailure`: Missing permissions

## Summary of Key Conventions

1. **Clean Architecture** — Strict layer separation with dependency rules
2. **BLoC Pattern** — Predictable state management with separate files
3. **Entity vs Model** — Clear distinction between business objects and DTOs
4. **Use Case Interface** — Single `call()` method for operations
5. **Widget Separation** — Screens use BLoC, widgets receive data via parameters
6. **Focusable Wrapper** — All interactive elements wrapped for keyboard navigation
7. **English Code & Comments** — Consistent language across codebase
